## はじめに

普段は Web や SaaS の世界で SRE / インフラエンジニアをやっている人にとって
行政におけるインフラ・ネットワークってどういう印象がありますでしょうか。
また「LGWAN」「ガバメントクラウド」「公共SaaS」っていうキーワードを聞いた時どう感じますでしょうか。

- なんとなく“レガシーで閉じてる世界”っぽい
- 「ガバメントクラウド」だけはなんとなく聞いたことあるけど、正体もよく分からない。それ以外は知らん。
- ほとんど SIer のオンプレのパッケージなんでしょ？
- 仕様書が PDF で 100 ページあって、そっと閉じたことがある

自分も最初はそんな感じでした。

でも、ちゃんと構造を追っていくと見え方が変わります。

- 行政インフラは「レガシー」じゃなくて **制約のめちゃくちゃ厳しい本番環境**
- そこで求められるのは、**アーキテクチャ設計の解像度**と **社会インパクトを意識した SRE 的判断**
- しかも、案件は突然降ってくるのではなく、**入札・調達仕様書という“起点”**から始まっている

この記事では、

> 「行政インフラ案件ってそもそもどう生まれて、調達仕様書にどんなことが書いてあって、どういうインフラの上で、それを満たすために LGWAN / ガバメントクラウドや公共 SaaS をどう選ぶのか？」

を、元Web系SREの1人の個人的な見解も踏まえて、ご紹介できればと思います。

## 2. そもそもオンプレ？クラウド？

まず、気になるのはオンプレミスなのかクラウドなのかという点です。
結論としては、**昔はオンプレミス前提でしたが、今は「クラウド前提」にシフトしている真っ最中**です。

現状を見ると、チャットやグループウェアなど、比較的新しいツールは導入時点でクラウドを活用しているケースが多く見られます
（LGTalk、LoGoチャット、Garoon、kintone など）。
一方で、昔からある基幹システム系は、SIer が構築・運用しているオンプレミス環境が残っていることが多いという印象です。

自治体や政府の情報システムは、少し前までは基本的にオンプレミス前提でした。

- 庁舎内やデータセンター内のサーバールームに物理サーバを設置する
- 5〜7 年サイクルでハード更新・OS 更新・基幹システムの更改を行う
- クラウドは「一部の周辺系で検討されるもの」程度の位置づけ

しかし、ここ数年で状況が大きく変わっています。

背景としてよく挙げられるのが、政府が打ち出している 「クラウド第一原則（クラウド・バイ・デフォルト）」 です。
政府情報システムの方式を検討するときに、クラウドサービスの採用を “第一候補（デフォルト）” としようという方針ですね。
これによって、従来の「自前で作る（Make）」文化から、「あるものを使う（Buy/Use）」文化への転換点に今の地方自治体は立たされているという状況です。

> （参考）政府情報システムにおけるクラウドサービスの適切な利用に係る基本方針（デジタル社会推進標準ガイドライン DS-310）2025 年（令和７年）X 月 XX 日 デジタル社会推進会議幹事会決定
> https://www.digital.go.jp/assets/contents/node/basic_page/field_ref_resources/44df7733-3df2-4e47-bf0b-85c0353c20c7/025b14b7/20250527_meeting_executive_policy_draft_04.pdf

さらに、今年は特に忙しそうにしています。
なぜなら、2025年度末（つまり、今年度末ですね）に向けて、標準化対象20業務をガバメントクラウドへ移行しなければならないというので、各自治体の方々やそのベンダーの方々がてんやわんやしている感じですね。

とはいえ、先ほどお伝えした通り、入札時の「調達仕様書」に「オンプレ前提」というシステムであれば、オンプレを使わなければならない場合も当然あります。
具体的には

1. 特定秘密・極秘情報レベルの機微情報を扱うシステム
2. 法令や業界規制的に、まだクラウド前提にできない領域
3. 超低レイテンシが要求される一部のリアルタイム系（現場装置に近いところ など）

などなど。

とはいえ、クラウド第一原則・ゼロトラストとか世の中の流れはそうなってきていると思っているので、自治体のシステムも今後2030年頃までに、5〜7 年サイクルとかのオンプレの既存システムがクラウドへ移行していくのではないかなと感じています。

## 1. その案件どこから来た？ ~ 入札・調達仕様書と“市場の生まれ方” ~

Web 企業だと、だいたいこんな感じでサービスが生まれますよね。

0. 既存市場やユーザー課題のリサーチをして、どこにニーズやポテンシャルがあるかを探る  
1. 社長や PdM、事業側が「こういうサービスをやりたい」と言う  
2. 企画・要件定義を一緒にやる  
3. エンジニアも最初から入って技術的な実現方法を詰める  
4. 実装して、試して、直していく  
5. うまくハマれば、そこに市場ができてくる

行政インフラ案件は、この「市場の生まれ方」と「最初のボールの投げられ方」がかなり違って見えます。

- まず行政側（自治体や国）の中で、  
  - 「この業務はもうアナログでは限界だ」  
  - 「制度改正に対応しないといけない」  
  - 「既存システムの更新時期が来た（ハード保守やリース、サポート終了など）」
  といった“業務起点の課題”が出てくる  
- その課題に対して、国の方針や標準仕様、予算メニューなどが整備されていく  
  - この段階で、業界団体やベンダーが意見を出したり、勉強会・ヒアリングを通じて要件形成に影響を与えようとする**ロビーイング的な動き**も入ってくる  
  - そうして「この領域にこういうシステムが必要です」という**市場の枠組み**が作られる  
- そのうえで各自治体が、自分たちの状況に合わせて、**「調達仕様書（要求仕様書）」**という形に落とし込む  
  - 調達仕様書を作成する前に、RFI（情報提供依頼書）を各事業者に求めて、情報収集することもあります。
    - e.g. [足立区文書管理システム再構築に関する情報提供依頼（RFI）](https://www.city.adachi.tokyo.jp/josys/20250930_bunsyorfi.html)
  - RFI で情報提供しないとこの案件が取れないというわけではないですが、この段階から手を挙げていった方が、その後の入札して自治体の案件を取れる確率は上がるかなと思います（特に、スタートアップは）
- その後、仕様書が公示され、そこで初めて事業者側から見ると「このテーマの市場が立ち上がった」「この条件で競争してください」という形で入札・提案のステージに出てくる

つまり、スタートアップが「サービスを作って市場をつくる」のとは逆で、  
行政の世界では **「政策・制度・予算 →（必要に応じてロビーイングを含む要件形成）→ 仕様書 → 入札を通じた市場」** という順番で  
案件と市場が立ち上がってくる傾向がある、というのが最初の大きな違いです。

:::details そもそも「入札」って何？

Web 系の世界で普通に仕事をしていると、「入札」ってあまり馴染みがないですよね。
でも、行政の世界では **システムを作る・買う = 入札する** という流れが基本です。

### 入札を Web 系エンジニアに例えると

ざっくり言うと、入札は **「公開オークション形式の外注」** みたいなものです。

```text
[民間企業の場合]
社長「この SaaS 作りたいから、A 社に発注しよう」
     → 社内稟議通れば OK、直接契約

[行政の場合]
自治体「このシステム作りたい。でも税金使うから…」
     → 「誰でも応募できる形で公開募集しないとダメ」
     → 「入札」という仕組みで、公平に事業者を選ぶ
```

### 入札の種類（ざっくり 3 パターン）

1. **一般競争入札（オープンな価格競争）**
   - 条件を満たせば誰でも参加できる
   - 基本的には「一番安い」事業者が勝つ
   - Web 的に言えば「GitHub Issues に要件書いて、一番安い見積もり出した人に発注」みたいな感じ

2. **企画競争入札（プロポーザル / コンペ方式）**
   - 価格だけじゃなく、**提案内容の良さ**で選ぶ
   - 「うちならこう作ります」「こういう体制でやります」みたいな提案書を出す
   - Web 的に言えば「RFP に対して、デザインや技術提案込みでプレゼンして、総合点で選ばれる」感じ
   - **行政のシステム案件は、こっちが多い**

3. **指名競争入札**
   - 自治体が「この会社たちの中から選ぶ」と事前に絞る
   - 実績がある事業者だけを指名して競わせる方式

### 入札のざっくりフロー

```text
[1] 自治体が「調達仕様書」を作成
    └─ 「こういうシステムが欲しい」「こういう要件を満たしてね」

[2] 公示（公開募集）
    └─ 「この案件やりたい事業者は応募してね」と公開される
    └─ 自治体の HP や入札情報サービスで見れる

[3] 事業者が応募・提案書提出
    └─ 仕様書を読み込んで、「うちならこう作ります」を書く
    └─ 見積もり金額も出す

[4] 審査・評価
    └─ 価格 + 提案内容 + 体制などを総合評価
    └─ 評価委員会みたいなのがスコアリングして決める

[5] 落札・契約
    └─ 勝った事業者と正式に契約
    └─ ここからやっと「開発スタート」
```

### Web 系エンジニアが知っておくべきポイント

- **案件は「公示」された時点で要件が大枠固まっている**
  - 「一緒に要件定義しましょう」じゃなくて、「この仕様書の通り作ってね」が先に来る
  - だから、仕様書を読み解く力がめちゃくちゃ大事
  - とはいえ、すでにある自治体の業務の大きな流れは地方自治法や地方財政法などで決まっているため、自治体毎に大きく異なることはないため、毎回0から作るわけではない。
  - すでに実績があって、サービス提供している自治体があれば、その自治体の時の落札した際の調達仕様書と見比べて、大きく異ならなければ、そのままそのシステムを導入する方針で
    - とはいえ、カスタマイズな要件はありますが、それは民間でもエンタープライズ案件だと、同じことかなと。

- **仕様書は「契約の根拠」になる**
  - Web 系の「とりあえず作って、あとで調整」が通用しにくい
  - 仕様書に書いてないことをやると追加費用交渉が発生したり、逆に書いてあることをやらないと契約違反になる

- **入札情報は誰でも見れる**
  - 各自治体の HP や、入札情報サービス（NJSS など）で公開されている
  - 「どんな案件が出てるか」を眺めるだけでも、行政インフラの世界観が分かる
:::

この仕様書の中に、機能要件だけじゃなくて、**非機能要件**や**ネットワーク・セキュリティの前提**が書かれています。

## 3. 調達仕様書のどこに「SRE が読むべきこと」が書いてあるのか

![inline](/images/af400c4d508a86/non-functional-requirements.png)

調達仕様書の構成はバラバラですが、だいたいこんな要素はよく出てきます。

- システムの目的・背景
- 業務ごとの機能要件
- 非機能要件（性能・可用性・セキュリティ・保守運用など）
- ネットワーク要件（LGWAN 接続・三層分離・端末配置など）
- 運用・監視・保守体制に関する要求
- 納期・体制・ドキュメント類

SRE / インフラエンジニア的には、特にこのあたりを「SLO っぽいもの」として読んでいきます。

- **環境**  
  - 「本番環境、研修環境、検証環境を用意すること」
  - 「研修環境は、本番と同等の環境を用意して、新人研修などでいつでも使えるようにしておくこと」 など
- **可用性**  
  - 「業務時間内の停止は原則認めないこと」  
  - 「月次での停止許容時間は◯時間以内とする」 など
- **性能**  
  - 「ピーク時同時接続◯ユーザでも◯秒以内に応答すること」  
  - 「日次バッチは業務開始までに完了すること」 など
- **セキュリティ / 監査**  
  - 「操作ログを◯年保管すること」  
  - 「利用者の操作履歴を追跡できること」 など
- **運用**  
  - 「24 時間 365 日の監視体制を敷くこと」  
  - 「重大障害発生時は◯分以内に一次報告を行うこと」 など

これって、ほぼそのまま **“日本語で書かれた SLO / 運用要件”** なんですよね。
Web の世界だと、SRE チームが自分たちで SLO を定義して、プロダクトと相談しながら決めていくことが多いと思います。

行政インフラの場合は、

> 「**先に SLO らしきものが仕様書側に書いてあって、それを達成するアーキテクチャを考える**」

という順番が逆になっている、くらいの感覚で読むと理解しやすいです。

## 3. 三層分離と α / α' / β / β' モデルを Web 系 SRE 的に整理する

突然ですが、仕様書を読んでいると

- 「三層分離」
- 「αモデル」「α'モデル」「βモデル」「β'モデル」

みたいな単語がちらほら出てきます。

### 3-1. 三層分離 = 超ざっくり言うと「トラストゾーンを 3 つに分ける」

三層分離の説明に入る前に、まず **LGWAN** について簡単に触れておきます。

LGWAN（Local Government Wide Area Network）は、**自治体や公的機関同士をつなぐ、全国スケールの閉域ネットワーク**です。Web系でいうと、インターネットではなく「でかい専用バックボーン」を持っていて、そこに各自治体が専用線や閉域VPNでぶら下がっているイメージです。住民記録や税務、福祉などの庁内業務システムは、このLGWAN経由で接続されることが一般的です。

（LGWANの詳細については、後述の「4-1. LGWAN：行政機関専用の閉域バックボーン」セクションも参照してください）

さて、三層分離は、Web3層構造の「プレゼンテーション層」「アプリケーション層」「データ層」のことか？と一瞬思ったりもするのですが、全く違います。笑

マイナンバー制度導入直前となる[2015年6月、日本年金機構が管理していた125万件もの個人情報が漏洩する事件](https://www.nenkin.go.jp/oshirase/topics/2015/20150721.files/press0820.pdf)をきっかけに、2016年に総務省の自治体セキュリティ強靭化ガイドラインで発表された「自治体情報システム強靭性向上モデル」におけるネットワーク分離モデルのことで **「マイナンバー利用事務系」「LGWAN接続系」「インターネット接続系」** の3層にネットワークを分離しましょう、という考え方です

| 層名                     | 主な用途・特徴                                            |
|--------------------------|--------------------------------------------------------|
| マイナンバー利用事務系    | マイナンバーを扱う、特にセンシティブな領域                 |
| LGWAN 接続系             | 住民記録・税・福祉など、自治体の庁内業務システム            |
| インターネット接続系     | Webブラウジング、メール、外部クラウドサービス利用など         |

基本的に、どの自治体の職員の方々もこの3層のどこかで業務をしています。その前提で「どの業務をどの層（どの接続系）でどの端末を使ってやるのか」に従って、自治体毎にいくつかのパターン（モデル）に分かれています。最初は、αモデルのみが定義されて（多分その頃はαモデルという言葉はなかったのかな？）その後、職員の利便性・生産性の向上やクラウド利用を目的としてガイドラインが改定され、いくつかのモデルが定義されていきました。

| モデル名                  | 概要                                                                                                       |
|--------------------------|-----------------------------------------------------------------------------------------------------------|
| **αモデル**   | 従来の形。LGWAN接続系に業務端末を置き、インターネットとは完全に分離する。                                        |
| **βモデル**              | 業務端末を「インターネット接続系」に配置するが、業務システムは「LGWAN接続系」に残すモデル。職員はインターネットを自由に使えるが、業務システムへは画面転送（VDI）等を使ってアクセスする必要がある。 |
| **β’モデル**             | 業務端末に加え、業務システム自体も「インターネット接続系（またはクラウド）」へ移行させるモデル。LGWAN接続系は最小限となり、フルクラウド化・ゼロトラストセキュリティを前提とした最も先進的な構成。 |
| **α'モデル**             | 2024年10月のガイドライン改定で正式に位置づけられたモデル。基本はαモデルの構成（三層分離）を維持しつつ、特定の安全なクラウドサービス（ISMAP登録済み等）に限り、LGWAN系から直接接続（ローカルブレイクアウト）することを許可する。 |

### 3-2. αモデル：一番堅くて、今も広く使われている基本形

αモデルは、三層分離の中でも「業務端末が LGWAN 接続系側」にある構成です。

- 職員の業務 PC は LGWAN 接続系にぶら下がる
- インターネットを見るときは、別端末やシンクライアントなどを経由する
- 各ネットワークは基本的に直接はつながない（USB 経由転送＋無害化など）

セキュリティ的には分かりやすくて堅い一方で、

- 端末を使い分けたり、
- ファイルをネットワーク越しに気軽に行き来させられなかったり、

現場の利便性はかなり犠牲になりがちです。

### 3-3. β / β' モデル：業務端末をインターネット側に出す「新三層分離」

そこで出てきたのが β / β' モデルです。

- **βモデル**
  - 業務システムは LGWAN 接続系に残す
  - 業務端末をインターネット接続系に置き、画面転送（VDI 等）で LGWAN 側のシステムを使う

- **β'モデル**
  - 業務システムも業務端末も、基本的にインターネット接続系側に出す
  - その代わり、エンドポイントセキュリティやゼロトラスト的な対策をかなりガチでやる

Web 的に雑に言えば、

- βモデル = 「インターネット側の端末から、閉域 VPC のシステムを VDI で操作する」
- β'モデル = 「もうシステムごとインターネット側に出して、エンドポイント＋クラウド側で守る」

というイメージです。

### 3-4. α'モデル：α の構造を残したまま、クラウドも使いたいという折衷案

α'モデルは、最近出てきた「αモデルを完全には崩さずに、クラウドも使いたい」という考え方です。

- 業務端末は LGWAN 接続系側に残したまま、
- 特定のクラウドサービスに対してだけローカルブレイクアウト（直接インターネット接続）する

という構成が例としてよく出てきます。

つまり、

> **「三層分離は維持しつつ、一部のクラウドだけ例外的に直でつなぐ」**

という少し器用なやり方です。

### 3-5. モデル名が仕様書に出てきたとき、SRE はこう読む

仕様書に

- 「三層分離 αモデル相当で構成すること」
- 「将来的に β / β' モデルへの移行を見据えること」
- 「α'モデルによるクラウドサービス利用を想定する」

みたいな文言が出てきたら、それは

> **ネットワークと端末配置の“前提条件”がすでに決まっている**

と読むと良いです。

Web の感覚で言えば、

- 「どの VPC に何を置くか」
- 「どこからどこへ直接通信できるか」

が先に決まっていて、その上にシステムを設計する、という感じです。

## 4. LGWAN / ガバメントクラウドをざっくり地図に乗せる

ここまでの前提（仕様書・三層分離・各モデル）を踏まえたうえで、  
LGWAN / ガバメントクラウドをマップに置いてみます。

### 4-1. LGWAN：行政機関専用の閉域バックボーン

LGWAN は、ざっくり言うと

> **自治体や公的機関同士をつなぐ、全国スケールの閉域ネットワーク**

です。

Web 系でいうと、インターネットではなく「でかい専用バックボーン」を持っていて、  
そこに各自治体が専用線や閉域 VPN でぶら下がっているイメージに近いです。

### 4-3. ガバメントクラウド：共通基盤としてのクラウド

ガバメントクラウドは、

> **複数のクラウド事業者上に用意された、政府向けの共通クラウド基盤**

と捉えると分かりやすいです。

- クラウド自体は AWS / GCP / Azure / OCI などのパブリッククラウド
- その上に、自治体向けの基幹システムや共通サービスを載せる
- セキュリティ・運用・調達の要件を満たすためのガイドラインが整備されている

## 5. 調達仕様書の一文をアーキテクチャに落とすまで

ここまで来ると、調達仕様書に書かれた一文を  
アーキテクチャにマッピングするイメージがつきやすくなります。

仮に、こんな仕様書の一文があったとします。

> 「本システムは、三層分離の αモデル相当の構成とし、  
>  LGWAN 経由で各自治体庁舎からアクセス可能であり、  
>  業務時間内のシステム停止は原則として認めないこと。」

これを SRE 的に分解すると：

- 三層分離 αモデル相当
  - 業務端末は LGWAN 接続系側
  - インターネットとの経路はかなり限定される
- LGWAN 経由でアクセス
  - ガバメントクラウドに置くなら、LGWAN 接続のための経路設計が必須
- 業務時間内は止められない
  - SLO 的には「業務時間内ほぼ 100% 稼働」が求められている
  - ローリングアップデートやメンテ時間をどう確保するかが設計ポイント

そこから導かれるアーキテクチャの方向性としては、

- ガバメントクラウド上でマルチ AZ 構成にするか？
- LGWAN 側とクラウド側の障害をどう分離して監視するか？

といった検討になります。

ここはまさに、**SRE の読解力・設計力が問われるポイント**です。

## 6. 公共 SaaS とは何者か？  
ガバメントクラウド上の「マルチテナント自治体 SaaS」という発想

最近よく出てくるのが「公共 SaaS」という言葉です。

雑にひとことで言うと、

> **「ガバメントクラウド上に構築された、自治体向けマルチテナント SaaS」**

です。

- 事業者がガバメントクラウド上にアプリケーションを構築する
- 複数の自治体が、そのアプリをマルチテナントで共同利用する
- 自治体側は「アプリ利用料 + ガバクラ利用料」を事業者にまとめて支払う（イメージ）

SaaS としてはわりと普通の前提ですが、

- **マルチテナント（テナントごとに別環境ではなく、共有リソースを使う）**
- **バージョンは共通（みんな同じバージョンを使う）**

といった要件が、ガバメントクラウドの側でも前提とされています。

Web 系から見れば、

> 「ようやく、自治体向けシステムも “ちゃんとした SaaS” として作る方向に振っていこうとしている」

と読むこともできます。

## 7. 「どこに構築するか」の決定木  
公共 SaaS / ガバメントクラウド / 自前クラウド

では、事業者側の SRE / インフラエンジニア視点で、

> 「この案件、どこにどう構築するのが良さそう？」

をざっくり考えるための思考の型を、テキスト決定木として書いてみます。

### 7-1. テキスト決定木（ドラフト）

```text
[Q1] 対象は自治体・公的機関向けの業務システムか？
  ├─ No
  │    → 一般民間向けサービス。
  │       普通に自前パブリッククラウドで OK。
  └─ Yes
       → [Q2] へ

[Q2] 調達仕様書・関連資料に
     「ガバメントクラウド利用」や
     「公共 SaaS」への言及があるか？
  ├─ Yes
  │    → 基本線はガバメントクラウド前提。 [Q3] へ
  └─ No
       → 自前クラウドも候補。 [Q5] へ

[Q3] このシステムは、
     複数自治体への横展開が前提か？
     （例：20〜数百団体レベル）
  ├─ Yes
  │    → [Q4] へ
  └─ No
       → 特定自治体専用 or ごく少数。
          ガバメントクラウド上に
          「単一自治体向けシステム」として構築
          （マルチテナントでなくてもよい）

[Q4] マルチテナント SaaS としての提供、
     公共 SaaS 要件を満たすための投資を
     ちゃんとやる覚悟はあるか？
  ├─ Yes
  │    → 公共 SaaS を目指して、
  │       ガバメントクラウド上にマルチテナント構築。
  │       将来的にマーケットプレイス的な展開も視野。
  └─ No
       → ガバメントクラウド上に
          各自治体ごとのシングルテナント環境を構築。
          （ガバクラは使うが SaaS にはしない）

[Q5] ガバメントクラウド指定はないが、
     LGWAN 接続や三層分離との整合性が
     強く求められているか？
  ├─ Yes
  │    → 選択肢は 2 つ：
  │       A) ガバメントクラウド上で構築
  │       B) 自前クラウド上に構築し、
  │          LGWAN などとの閉域接続を設計
  │    → 「どちらが運用・セキュリティ設計を
  │        シンプルにできるか？」で判断。
  └─ No
       → 自前パブリッククラウド上で構築で OK。
          LGWAN 連携が不要 or ごく限定的なら、
          ベンダ側クラウドの自由度を優先しがち。
````

### 7-2. 自前クラウド寄りになるパターン

現実的には、こんな要素があると「自前パブリッククラウド」のほうに寄りがちです。

* 自治体だけでなく、民間企業にも提供したいサービス
* すでに自前クラウド上で SaaS を持っていて、それをカスタマイズして出したい
* テナントごとの差分が大きく、マルチテナント前提にしづらい

この場合、

* コア部分は自前クラウド
* LGWAN や庁内ネットワークとの接続用に、ガバクラ側にゲートウェイだけ置く

みたいなハイブリッド構成も選択肢になります。

### 7-3. ガバメントクラウド寄りになるパターン

逆に、次のような案件は「ガバメントクラウドでやった方が筋がいい」領域です。

* 標準化対象業務（住民情報・税・福祉など）の基幹システム
* 多数の自治体に同じ仕様で展開したい共通サービス
* LGWAN からの利用が前提で、閉域ネットワーク設計が重たい

この場合、ガバメントクラウド上に

* マルチテナントの公共 SaaS として作るのか
* 自治体ごとの環境を作るのか

を決めるのが、アーキテクチャ検討の大きな分岐になります。

## 8. 制約だらけだからこそ、SRE が燃えるフィールド

ここまで見てきたように、行政インフラの世界は、

* ネットワークや端末配置（α / β / α' / β' モデル）
* LGWAN / ガバメントクラウド / 公共 SaaS
* 調達仕様書に書かれた、たくさんの非機能要件・運用要件

といった前提条件と制約でガチガチです。

でも、SRE 的に言うと、これはかなり **“おいしい本番環境”** でもあります。

* SLO / エラーバジェット / Postmortem のような考え方は、そのまま効く
* 非機能要件を設計に翻訳する力が、ダイレクトに価値になる
* 「1 時間落ちたら何千人の住民が手続きできない」といった、**社会インパクトが直結する世界**

たしかに、Web 系のように「とりあえずデプロイして試してみる」みたいなノリでは動けません。

* 変更管理も厳しい
* ドキュメントも多い
* 監査もある

でもそれは裏を返せば、

> **“ちゃんと考え抜かれたアーキテクチャ”と
> “安全な自動化・運用プロセス”が評価される世界**

だとも言えます。

## おわりに：レガシーではなく、「制約の厳しい本番環境」としての行政インフラ

この記事では、あえて

* 入札・調達仕様書という「案件の入り口」
* 非機能要件の読み方
* 三層分離と α / α' / β / β' モデル
* LGWAN / ガバメントクラウド
* 公共 SaaS と、構築場所の決定木

といった順番で、行政インフラの世界をざっくり一周してみました。

伝えたかったのは、次の 3 つです。

1. 行政インフラはレガシーではなく、**制約のめちゃくちゃ厳しい本番環境**だということ
2. Web 系 SRE / DevOps のスキルは、**ほぼそのまま（むしろ濃く）活きる**ということ
3. その設計・運用が、**技術的なおもしろさと社会インパクトを同時に生む**フィールドだということ

もしこの記事を読んで、

* 「αモデルってそういうことか」
* 「公共 SaaS って、要するにガバクラ上のマルチテナント SaaS ね」
* 「仕様書を SLO とアーキ要件として読み替えるの、ちょっと面白そうかも」

と思ってもらえたなら、GovTech / 行政インフラの世界に興味を持つきっかけとしては大成功です。

どこかで一緒に、制約だらけの本番環境で遊べる日が来たらうれしいです。
